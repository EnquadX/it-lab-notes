# OpenClaw Setup — 01 Early-Stage Setup

## Why I started this
I wanted to build a proper personal assistant setup that I would actually use, not just another AI tool that seems interesting for a few days and then gets ignored.

A big part of that was making it fit into the way I already work, instead of forcing myself into some workflow that looks good on paper but is annoying in real life.

## Why I bought the Mac mini for this
I bought the Mac mini because I am already deep in the Apple ecosystem, so it made more sense to give OpenClaw a home that fits the rest of my setup.

I could have kept trying to force this onto a different machine, but I wanted something:
- always on
- low friction
- quiet
- stable
- easy to use alongside the rest of my Apple devices

That mattered more than just “can it run OpenClaw?”.

## What I did first
The early stage was mainly about getting the machine ready and getting OpenClaw running in a way that felt usable.

Main steps:
- prepared the Mac mini as the host machine
- installed Homebrew
- installed OpenClaw
- started building the assistant context properly instead of leaving it as a blank chatbot

I also added a strong initial prompt/context layer about me so the assistant had a better starting point.
That mattered because I did not want to repeat the same background over and over again every time.

## Technical things I learned early
A few things became obvious quite quickly:

### 1. The host machine matters
The machine is not just where the software runs.
It affects:
- reliability
- how often I will actually use it
- what local apps it can interact with
- how well it fits into my day-to-day workflow

### 2. Apple-native access is a big advantage here
Running this on a Mac made more sense because later on it could work with things like:
- Reminders
- Calendar
- Notes
- Mail
- Finder

That made the setup feel more like a real assistant and less like a toy.

### 3. Context matters a lot
A raw AI model is not the same thing as an assistant.
The assistant only started becoming more useful once I began giving it:
- background about me
- rules
- structure
- memory files
- clear expectations for how it should work

### 4. Friction kills adoption
Even early on, it was obvious that if the setup is annoying, awkward, or hard to maintain, I will stop using it properly.
So a lot of the early choices were really about reducing friction.

## What this stage was really about
This stage was not about advanced features.
It was about getting the foundations right:
- choosing the right host
- installing the core tools
- setting up the first layer of context
- making sure the whole thing could grow into something more useful later

## Early lesson
One of the main things I noticed early was that this works much better when it is treated like infrastructure, not like a novelty.

The more I gave it a proper home, proper structure, and proper context, the more it started feeling like something I could actually use.

## Current status
This is still the early setup stage, but it set the base for everything that came after.
